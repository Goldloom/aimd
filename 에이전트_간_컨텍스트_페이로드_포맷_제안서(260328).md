# 에이전트 간 컨텍스트 페이로드 포맷 제안서

## 문서 목적

이 문서는 여러 에이전트가 협업할 때, 작업 맥락과 제약사항과 다음 행동을 일관된 방식으로 주고받기 위한 `에이전트 간 컨텍스트 페이로드 포맷`을 제안한다.

핵심 문제는 단순하다.

- 에이전트는 도구에 연결될 수 있다
- 에이전트는 다른 에이전트에게 태스크를 넘길 수 있다
- 하지만 에이전트가 서로 무엇을, 어떤 우선순위로, 어떤 제약 아래, 어디까지 검증된 상태로 넘겨야 하는지에 대한 콘텐츠 포맷 표준은 비어 있다

이 문서는 그 빈 공간을 메우는 최소 제안을 목표로 한다.

---

## 배경

현재 생태계에는 이미 중요한 조각들이 존재한다.

- `AGENTS.md` 계열 문서: 에이전트에게 프로젝트 규칙과 작업 방식을 알려주는 관행이 빠르게 확산되고 있다
- `MCP`: 에이전트가 도구와 데이터 소스에 접근하는 연결 규약을 제공한다
- `A2A`: 에이전트끼리 작업을 넘기고 협력하는 프로토콜 계층을 다룬다
- Markdown 기반 skill file, runbook, prompt file: 실제 현장에서는 이미 Markdown 파일이 사실상의 에이전트 운영 매뉴얼 역할을 하고 있다

하지만 이것들은 주로 아래를 다룬다.

- 어떻게 연결할 것인가
- 어떤 도구를 사용할 수 있는가
- 어느 파일을 읽어야 하는가

반대로 아직 비어 있는 것은 이것이다.

- 에이전트가 다른 에이전트에게 어떤 맥락 묶음을 전달해야 하는가
- 그 안에서 어떤 항목이 필수이고 어떤 항목이 참고인지 어떻게 구분할 것인가
- 무엇이 검증되었고 무엇이 가설인지 어떻게 표시할 것인가
- 어떤 부분은 수정 금지이고 어떤 부분은 다음 에이전트가 이어서 작업해야 하는지 어떻게 표시할 것인가

오늘의 많은 핸드오프는 사실상 서술형 메시지에 의존한다. 이 방식은 빠르지만, 누락과 충돌과 재해석 비용이 크다.

따라서 필요한 것은 새 프로토콜이 아니라, 프로토콜 위를 흐르는 `에이전트용 콘텐츠 포맷`이다.

---

## 제안 요약

제안하는 포맷은 다음과 같은 성격을 가진다.

- 사람도 읽을 수 있고 에이전트도 안정적으로 파싱할 수 있어야 한다
- 우선순위, 제약, 검증 상태, 참조, 다음 행동을 명시해야 한다
- 기존 Markdown 기반 워크플로와 호환되어야 한다
- 필요한 경우 JSON 같은 정규화된 구조로도 변환 가능해야 한다

이 문서에서는 이 포맷을 임시로 `ACP`라고 부른다.

`ACP = Agent Context Payload`

ACP는 프로토콜이 아니다. ACP는 내용물이다.

- `MCP`와 `A2A`는 파이프
- `ACP`는 파이프 안을 흐르는 컨텍스트 페이로드

---

## 포지셔닝

이 제안은 기존 AIMD 논의를 다음처럼 정리한다.

- `AIMD`: 사람이 AI를 통해 생성하는 AI-generated structured Markdown
- `ACP`: 에이전트 간 전달에 최적화된 정제된 컨텍스트 페이로드

즉, AIMD는 에이전트들이 공유·참조·갱신하는 구조 문서이고, ACP는 그 문서에서 핸드오프 시점에 추출되는 전달용 페이로드다.

이 구분이 중요한 이유는 다음과 같다.

- 구조 문서와 전달 페이로드를 분리하면 개념이 선명해진다
- `.aimd` 같은 확장자와 편집기 기능은 계속 유지할 수 있다
- 사람은 AIMD 생성을 요청하고 저장 위치를 선택하면 된다
- 생성 이후에는 여러 AI 에이전트가 같은 AIMD를 참조하며 `:::` 구조층을 갱신할 수 있다
- 실제 핸드오프 시점에는 `:::human` 제거, `inherits` 병합, `@ref` 정리 등 페이로드 조립 규칙을 명확히 적용할 수 있다

한 문장으로 정리하면 다음과 같다.

> AIMD는 human-directed, AI-generated shared structured layer이고, ACP는 handoff layer이다.

## 문제 정의

에이전트 간 협업에서 반복적으로 발생하는 손실은 대체로 아래 다섯 가지다.

### 1. 중요도 손실

어떤 내용이 절대 지켜야 하는 규칙인지, 단순 참고인지 구분되지 않는다.

### 2. 검증 상태 손실

수치, 사실, 해석 중 무엇이 검증 완료이고 무엇이 아직 가설인지 드러나지 않는다.

### 3. 변경 금지 범위 손실

이전 에이전트가 고정한 산출물과, 다음 에이전트가 수정 가능한 범위가 분리되지 않는다.

### 4. 출처 손실

주장이 어디서 왔는지, 어떤 파일과 어떤 에이전트가 근거인지 빠르게 추적하기 어렵다.

### 5. 다음 행동 손실

핸드오프가 끝난 뒤 수신 에이전트가 정확히 무엇을 이어받아야 하는지 모호해진다.

ACP는 이 다섯 가지 손실을 줄이기 위한 최소 구조를 제공한다.

---

## 설계 원칙

### 1. 최소 구조, 최대 휴대성

복잡한 스키마보다, 다양한 모델과 도구가 바로 이해할 수 있는 얇은 구조를 우선한다.

### 2. 서술보다 선언

자연어 설명은 남기되, 우선순위와 상태와 제약은 구조적으로 표시한다.

### 3. 검증 상태의 명시

`VERIFIED`, `ASSUMPTION`, `OPEN`처럼 상태를 분리해서, 수신 에이전트가 재계산과 재검증 범위를 판단할 수 있어야 한다.

### 4. 수정 가능 범위의 명시

`FREEZE`와 `UNFREEZE` 같은 규칙을 통해 산출물 잠금 범위를 표현한다.

### 5. 참조 우선

길게 복붙하는 대신 `@ref` 같은 참조를 통해 원문과 근거 위치를 연결한다.

### 6. 사람이 검토 가능해야 함

포맷은 기계 친화적이되, 사람이 diff와 review를 할 수 있어야 한다.

---

## 최소 페이로드 모델

ACP는 크게 두 층으로 구성된다.

### 1. 헤더

페이로드 메타데이터를 담는다.

필수 권장 항목:

- `acp`: 포맷 버전
- `id`: 페이로드 식별자
- `from`: 발신 에이전트
- `to`: 수신 에이전트 또는 역할
- `task`: 현재 태스크 식별자
- `created_at`: 생성 시각
- `ttl`: 유효 기간

선택 항목:

- `thread`
- `project`
- `source`
- `confidence`
- `visibility`

### 2. 블록

실제 컨텍스트를 담는다.

최소 권장 블록:

- `:::summary` 현재 상태 요약
- `:::ai` 우선순위, 제약, 금지, 잠금 범위
- `:::refs` 파일/문서/코드 참조
- `:::open` 미해결 이슈
- `:::next` 다음 에이전트가 해야 할 일

선택 블록:

- `:::decision` 확정된 결정
- `:::evidence` 검증 근거
- `:::artifact` 생성된 산출물
- `:::diff` 이전 상태 대비 변경점

---

## 핵심 지시자

ACP에서 특히 중요한 지시자는 아래와 같다.

### `FROM`

이 컨텍스트를 만든 주체를 나타낸다.

예:

`FROM: research_agent`

### `VERIFIED`

검증이 끝난 사실, 수치, 계산, 판단을 나타낸다.

예:

`VERIFIED: 주문 건수 집계 SQL, 매출 표 숫자, 환율 적용 방식`

### `ASSUMPTION`

아직 사실로 확정되지 않은 가정을 나타낸다.

예:

`ASSUMPTION: webhook retry interval은 30s일 가능성이 높음`

### `CRITICAL[n]`

반드시 지켜야 하는 제약이다. 숫자가 낮을수록 우선순위가 높다.

예:

`CRITICAL[1]: 아래 금액 재계산 금지`

### `FREEZE`

후속 에이전트가 수정하면 안 되는 범위를 지정한다.

예:

`FREEZE: executive_summary`

### `NEXT`

다음 책임 주체를 명시한다.

예:

`NEXT: writing_agent`

## 기본 직렬화 제안

ACP는 Markdown 기반 직렬화를 기본 제안으로 삼는다. 이유는 간단하다.

- 사람이 읽을 수 있다
- Git diff가 잘 된다
- 기존 문서 시스템과 충돌이 적다
- AIMD와 자연스럽게 연결된다

예시:

```markdown
---
acp: "0.1"
id: handoff-payment-042
from: research_agent
to: writing_agent
task: payment_race_condition_report
created_at: 2026-03-28T10:30:00+09:00
ttl: 24h
project: commerce-platform
source: docs/payment/payment.aimd
---

:::summary
결제 경쟁 상태 원인 분석 완료.
핵심 원인은 checkout finalize 단계의 중복 호출과 idempotency key 저장 타이밍 불일치.
:::

:::ai
FROM: research_agent
VERIFIED: 장애 재현 절차, DB write 순서, 중복 결제 발생 조건
CRITICAL[1]: 아래 금액 계산 예시는 재계산하지 말 것
CRITICAL[2]: executive_summary 문체와 길이 유지
FREEZE: executive_summary
NEXT: writing_agent
:::

:::refs
@ref:docs/payment/payment.aimd
@ref:src/order/checkout.ts::CheckoutService.finalize
@ref:src/payment/payment.service.ts:L84-L160
:::

:::open
- idempotency key TTL 운영값은 문서 근거 추가 필요
- retry backoff 정책은 운영 환경별 차이 검증 필요
:::

:::next
1. executive summary 문단화
2. 운영 영향도 섹션 작성
3. open 항목은 "추가 검증 필요"로 명시
:::
```

이 예시는 자연어만으로 전달하던 핸드오프를, 우선순위와 잠금 범위와 다음 행동까지 포함한 구조적 페이로드로 바꾼다.

## 권장 처리 규칙

수신 에이전트는 ACP를 받을 때 최소한 아래 순서로 해석해야 한다.

1. 헤더를 읽고 발신자, 수신자, 태스크, 유효 기간을 확인한다
2. `:::ai` 블록을 가장 먼저 읽는다
3. `CRITICAL[n]`을 우선순위 순으로 적용한다
4. `FREEZE` 지정 범위를 수정 대상에서 제외한다
5. `VERIFIED`와 `ASSUMPTION`을 구분한다
6. `@ref`를 통해 원문과 근거를 조회한다
7. `:::open`과 `:::next`를 기준으로 후속 작업 범위를 정한다

이 처리 순서가 중요한 이유는, 같은 텍스트라도 어떤 항목을 먼저 읽느냐에 따라 결과가 달라지기 때문이다.

---

## 병합 규칙 제안

ACP는 단일 메시지뿐 아니라 누적 핸드오프에도 쓰일 수 있다. 이를 위해 최소 병합 규칙이 필요하다.

### 1. 최신 페이로드 우선

같은 `task`와 같은 블록 유형에 대해 충돌이 날 경우, 더 최신 `created_at`을 가진 페이로드를 우선한다.

### 2. `CRITICAL`은 누적

부모 지시와 새 지시는 기본적으로 누적한다. 동일 번호라도 덮어쓰기보다는 병합을 우선한다.

### 3. `FREEZE`는 명시적 해제 전까지 유지

`FREEZE`는 후속 에이전트의 자유 재해석보다 우선한다. 해제는 `UNFREEZE`가 있을 때만 허용한다.

### 4. `VERIFIED`는 근거와 함께 유지

검증 완료 항목은 출처 또는 참조가 함께 있어야 한다. 근거가 없으면 `VERIFIED`보다 낮은 상태로 취급한다.

### 5. `OPEN`은 닫기 전까지 살아 있다

해결되지 않은 열린 항목은 후속 페이로드에서도 명시적으로 닫히기 전까지 유지한다.

## AIMD와의 관계

ACP를 독립 개념으로 두더라도, 실제 작성과 운영에서는 AIMD와 자연스럽게 결합된다.

### 권장 구조

- 사람이 제공하는 입력: 자연어 요청 또는 자유서술 Markdown / PRD
- AI가 생성하는 구조 문서: `.aimd`
- 에이전트 핸드오프 시 생성되는 결과물: `ACP`

예를 들면 다음과 같은 흐름이다.

1. 사람은 PRD나 요구사항을 자연어 또는 Markdown으로 제공한다
2. AI는 이를 바탕으로 `.aimd` 파일을 생성하고, 사람은 저장 위치만 결정한다
3. 여러 AI 에이전트는 같은 AIMD를 참조하며 `:::intent`, `:::flow`, `:::api`, `:::ai` 같은 구조층을 갱신한다
4. 핸드오프 시 필요한 부분만 추출하고 정규화해 ACP로 전달한다

이 흐름에서 AIMD는 AI 에이전트들의 공유 작업 문서이고, ACP는 실행 시점의 전달 묶음이다.

중요한 점은 `:::` 블록이 반드시 인간을 위한 표현이 아니라는 것이다.

- 사람은 AIMD 생성을 지시하고 결과를 검토할 수 있다
- 그러나 `:::` 구조층 자체는 주로 AI가 읽고 업데이트하는 운영 레이어다
- 따라서 인간은 자유서술 원문이나 최종 산출물을 중심으로 보고, 구조층 전체를 직접 읽을 필요는 없다

---

## 시스템 프롬프트 요구사항

현재 모델에서 AIMD의 성공 여부는 포맷 정의만으로 결정되지 않는다. 실제로는 `시스템 프롬프트가 AI에게 AIMD를 어떻게 생성하고 갱신하라고 가르치느냐`가 훨씬 중요하다.

왜냐하면 이 구조에서는:

- 사람이 AIMD 생성을 직접 문법으로 수행하지 않는다
- 사람이 AI에게 AIMD 생성을 요청한다
- 이후 여러 AI 에이전트가 같은 AIMD를 참조하며 `:::` 구조층을 갱신한다

즉, 시스템 프롬프트는 사실상 `AIMD 운영 규약` 역할을 한다.

### 시스템 프롬프트가 반드시 알려줘야 하는 것

1. AIMD의 정의

- AIMD는 사람이 AI를 통해 생성하는 AI-generated structured Markdown이다
- AIMD는 공유 구조 문서이고, ACP는 그 문서에서 추출되는 handoff payload다

2. 입력 모델

- 사용자는 자연어 요청, PRD, 자유서술 Markdown, 기존 문서를 줄 수 있다
- 사용자가 `.aimd`를 직접 타이핑하지 않아도 된다
- AI는 입력을 바탕으로 `.aimd` 파일을 생성하거나 갱신해야 한다

3. 책임 분담

- 사람은 의도, 제약, 승인, 저장 위치를 결정한다
- AI는 AIMD 구조와 `:::` 블록 내용을 생성한다
- 생성 이후 AI 에이전트들은 같은 AIMD를 참조하고 필요한 구조층을 업데이트한다

4. `:::` 블록의 성격

- `:::` 블록은 인간 문서 서식보다 AI 운영 레이어에 가깝다
- 따라서 시스템 프롬프트는 이 블록을 AI가 생성·유지해야 한다고 명시해야 한다
- 인간은 결과를 검토할 수 있지만, 구조층 전체를 직접 관리할 필요는 없다

5. 사실성과 불확실성 구분

- 사용자 제공 사실, 코드/문서 기반 사실, AI 추론을 구분해야 한다
- 불확실한 내용은 `OPEN`, `ASSUMPTION`, `ASK`, `TODO` 같은 상태로 표시해야 한다
- 검증된 사실만 `VERIFIED`로 남겨야 한다

6. 협업 규칙

- 기존 AIMD가 있으면 먼저 읽고 점진적으로 갱신한다
- 다른 에이전트가 기계적으로 읽을 수 있게 구조를 안정적으로 유지한다
- 역할별 handoff가 필요할 때는 AIMD 전체가 아니라 ACP를 추출한다

### 최소 필수 지시자

시스템 프롬프트는 적어도 아래 지시자를 AI가 일관되게 이해하게 해야 한다.

- `CRITICAL[n]`: 절대 어기면 안 되는 제약
- `VERIFIED`: 근거가 있는 확인 완료 정보
- `OPEN`: 아직 닫히지 않은 문제
- `FREEZE`: 후속 에이전트가 함부로 변경하면 안 되는 범위
- `NEXT`: 다음 역할 또는 다음 작업

### 실무적 의미

시스템 프롬프트가 약하면 AIMD는 단순한 예쁜 문서 포맷으로 끝난다.

반대로 시스템 프롬프트가 명확하면:

- AI가 AIMD를 일관되게 생성한다
- 에이전트들이 같은 문서를 안정적으로 재사용한다
- `:::` 구조층이 실제 handoff memory로 작동한다

따라서 AIMD/ACP 제안에서 시스템 프롬프트는 부가 요소가 아니라 필수 인프라에 가깝다.

## 디버깅 및 감사 가능성

이 구조의 중요한 장점 하나는, 나중에 개발 결과가 기대와 다를 때 `AI가 무엇을 어떻게 잘못 판단했는지`를 추적하기 쉬워진다는 점이다.

기존에는 문제가 생기면 보통 아래와 같은 상태가 된다.

- 왜 이런 구현이 나왔는지 알기 어렵다
- 어느 에이전트가 잘못 해석했는지 추적이 어렵다
- 요구사항 오해인지, 사실 오판인지, handoff 실패인지 구분이 안 된다

반대로 AIMD의 `:::` 구조층과 ACP를 함께 운영하면, 실패 원인을 훨씬 더 구조적으로 볼 수 있다.

예를 들어 확인 가능한 항목은 다음과 같다.

- 어떤 요구사항이 `CRITICAL`로 명시됐는데도 후속 에이전트가 무시했는가
- 어떤 정보가 실제로는 불확실했는데 `VERIFIED`처럼 다뤄졌는가
- `OPEN`으로 남겨야 할 리스크가 중간 단계에서 사라졌는가
- `FREEZE`된 계약을 누가 바꿨는가
- 어느 시점의 ACP handoff에서 잘못된 전제가 넘어갔는가

즉, `:::` 레이어는 단순 컨텍스트 저장 공간이 아니라 `AI 판단의 구조화된 흔적`으로도 작동한다.

### 실무적 의미

이 특성은 단순히 문서를 더 예쁘게 만드는 문제가 아니다.

- 오류 원인 분석이 빨라진다
- 다음 에이전트가 같은 오판을 반복할 가능성이 줄어든다
- 사람도 AI의 판단 근거를 검토할 수 있다
- 멀티 에이전트 환경에서 책임 경계와 handoff 실패 지점을 찾기 쉬워진다

### 왜 이게 중요한가

에이전트 협업에서 진짜 어려운 것은 `한 번에 잘 만드는 것`만이 아니다. `잘못되었을 때 왜 잘못되었는지 설명 가능한 구조`를 갖는 것이 더 중요하다.

그 점에서 AIMD의 `:::` 구조층과 ACP는 다음 역할을 동시에 수행한다.

- 컨텍스트 전달 레이어
- handoff memory 레이어
- 디버깅 레이어
- 감사 가능성 레이어

### 엔터프라이즈 감사/컴플라이언스 관점

이 특성은 엔터프라이즈 환경에서 더 큰 의미를 가진다. 대기업, 금융, 헬스케어, 공공 영역에서는 단순히 `AI가 잘 만들었는가`보다 `왜 그렇게 판단했는가`, `누가 무엇을 넘겼는가`, `어떤 제약이 적용됐는가`가 더 중요할 때가 많다.

이때 AIMD와 ACP는 다음 가치를 제공할 수 있다.

- 어떤 에이전트가 어떤 전제를 바탕으로 작업했는지 추적 가능
- 어떤 항목이 `VERIFIED`였고 어떤 항목이 `OPEN` 상태였는지 기록 가능
- `FREEZE`된 계약이나 제약이 언제, 누구에 의해 변경되었는지 확인 가능
- handoff 시점의 구조화된 판단 흔적을 감사 로그처럼 검토 가능

즉, 이 구조는 단순한 생산성 도구를 넘어서 `AI 작업 과정의 설명 가능성`과 `감사 가능성`을 높인다.

규제가 강한 조직에서는 바로 이 점이 중요하다.

- 결과물만 있으면 부족하다
- 판단 경로와 제약 적용 이력이 필요하다
- 나중에 문제 발생 시 책임 경계와 승인 흐름을 설명할 수 있어야 한다

그 점에서 AIMD의 `:::` 구조층과 ACP는 엔터프라이즈 AI 운영에서 `감사 가능한 협업 기록`으로 기능할 수 있다.

따라서 이 제안의 가치는 단순한 포맷 설계가 아니라, `AI 협업의 실패 원인을 추적 가능하게 만드는 운영 구조`를 제공한다는 데 있다.

## MCP, A2A, AGENTS.md와의 관계

ACP는 기존 표준을 대체하려는 것이 아니다. 오히려 빈 부분을 메우려는 것이다.

### `AGENTS.md`

`AGENTS.md`는 프로젝트 단위의 상시 지침 문서에 가깝다. ACP는 특정 핸드오프 단위의 동적 컨텍스트 묶음이다.

### `MCP`

MCP는 도구와 데이터 접근을 표준화한다. ACP는 그 도구를 통해 참조하거나 전달할 실제 컨텍스트 내용을 구조화한다.

### `A2A`

A2A는 에이전트 간 통신과 태스크 교환의 프로토콜 계층이다. ACP는 그 메시지에 첨부되거나 포함될 수 있는 콘텐츠 페이로드다.

정리하면:

- `AGENTS.md`: 장기 정책
- `AIMD`: AI-generated structured shared document
- `ACP`: 핸드오프 내용물
- `MCP/A2A`: 전달 경로

---

## 왜 지금 필요한가

지금은 에이전트 활용이 실험을 넘어 운영 단계로 넘어가는 시기다. 이 시기에는 다음 문제가 동시에 커진다.

- 한 에이전트가 한 번에 모든 일을 하지 않는다
- 역할 분화와 핸드오프가 늘어난다
- 검토, 요약, 작성, 코드 수정, 배포 같은 작업이 분리된다
- 장기 메모리와 런타임 컨텍스트의 경계가 중요해진다

따라서 `도구 연결 표준`만으로는 부족하고, `컨텍스트 전달 표준`이 필요해진다.

ACP는 이 변화에 대해 비교적 가볍고 현실적인 출발점이 될 수 있다.

## 핵심 페르소나

ACP/AIMD의 대표 사용 시나리오 중 하나는 `하나의 PRD를 여러 AI 에이전트가 동시 또는 순차적으로 참조하며 개발하는 환경`이다.

이 경우 역할은 보통 다음처럼 나뉜다.

- PRD 작성 및 우선순위 정리
- 아키텍처 초안 생성
- 프론트엔드 / 백엔드 / 데이터 작업 병렬 수행
- 리뷰 / QA / 배포 검증 순차 수행

이 흐름에서 AIMD는 사람이 AI를 통해 생성하고, 이후 여러 에이전트가 함께 참조·갱신하는 `PRD의 구조화된 공통 문서`로 작동한다. ACP는 그 문서에서 역할별로 추출되는 `핸드오프 페이로드`다.

### 1. Product Manager / Product Owner

이 페르소나는 PRD를 작성하고, 기능 목표와 범위와 우선순위를 정의한다.

주요 문제:

- 같은 PRD를 여러 에이전트가 제각각 해석한다
- 필수 요구사항과 참고 요구사항이 섞여 전달된다
- 사람이 읽는 PRD와 에이전트가 받는 작업 지시가 분리된다

가치:

- PM/PO는 `이 PRD를 AIMD로 생성해줘`라고 요청하는 방식으로 구조 문서를 만들 수 있다
- ACP로 변환하면 같은 PRD에서 에이전트별 컨텍스트 페이로드를 분기할 수 있다
- 결과적으로 `하나의 PRD를 여러 에이전트가 공유하되, 각자 필요한 맥락만 정확히 받는 구조`를 만들 수 있다
- 사람은 AIMD 파일의 저장 위치와 검토 여부를 결정하고, 구조 블록의 상세 갱신은 AI가 맡을 수 있다

### 2. AI Tech Lead / Agent Orchestrator

이 페르소나는 여러 에이전트에게 작업을 분배하고, 병렬 또는 순차 실행 흐름을 설계한다.

주요 문제:

- 어떤 제약이 전역 규칙인지, 특정 작업에만 적용되는지 구분이 어렵다
- 병렬 작업 중 가정과 결론이 충돌한다
- 이전 에이전트가 검증한 내용을 다음 에이전트가 다시 계산한다

가치:

- ACP의 `CRITICAL[n]`, `VERIFIED`, `FREEZE`, `NEXT`는 멀티 에이전트 오케스트레이션에 직접적인 제어 수단을 제공한다
- 병렬 에이전트에는 같은 PRD 원문 대신 역할별 ACP를 나눠 전달할 수 있다
- 순차 핸드오프에서는 이전 단계의 검증 결과와 수정 금지 범위를 다음 단계에 안정적으로 넘길 수 있다

### 3. Specialist Implementer / Reviewer / QA Agent

이 페르소나는 프론트엔드, 백엔드, 인프라, 테스트, 리뷰 중 특정 작업만 담당한다.

주요 문제:

- 전체 PRD를 다 읽어야 해서 잡음이 많다
- 현재 자신이 바꾸면 안 되는 부분이 무엇인지 모른다
- 미해결 이슈와 확정 이슈가 혼재되어 있다

가치:

- ACP는 역할에 맞는 최소 컨텍스트만 전달할 수 있다
- `OPEN`, `VERIFIED`, `FREEZE`로 현재 상태를 빠르게 구분할 수 있다
- QA나 리뷰 단계에서는 무엇이 이미 합의되었고 무엇이 추가 검증 대상인지 명확해진다

### 4. Enterprise AI Platform Owner / Engineering Manager

이 페르소나는 조직 차원에서 에이전트 협업을 통제하고, 감사 가능성과 표준화를 중요하게 본다.

주요 문제:

- 프로젝트마다 handoff 방식이 다르다
- 누가 어떤 맥락을 넘겼는지 추적이 어렵다
- 규정 준수와 승인 체계를 agent workflow에 녹이기 어렵다

가치:

- ACP는 agent handoff를 조직 공통 포맷으로 만들 수 있다
- AIMD는 장기 문서 자산과 런타임 실행 문맥을 연결해준다
- 결과적으로 팀별 문서 관행을 넘어서 `조직 차원의 에이전트 운영 규율`을 만들 수 있다

## PRD 기반 병렬 개발 시나리오

ACP/AIMD가 특히 유용한 장면은 `하나의 PRD를 기준으로 여러 AI 에이전트가 동시에 구현하고, 이후 순차적으로 통합과 검증을 이어가는 개발 흐름`이다.

예를 들어 `결제 실패 재시도 기능`을 개발한다고 가정해보자.

Product Manager는 요구사항, 범위, 예외 조건, 비즈니스 우선순위를 자연어 또는 Markdown으로 제공하고, AI에게 `이 PRD를 AIMD로 생성해줘`라고 요청한다. AI는 이를 `.aimd`로 생성하고, 사람은 그 파일을 둘 폴더 위치를 결정한다. 이후 에이전트들은 같은 AIMD를 참조하며 `:::` 구조층을 업데이트한다.

전체 흐름은 아래처럼 볼 수 있다.

```text
[PM / PO]
  prompt 또는 payment-retry-prd.md
  "이 PRD를 AIMD로 생성해줘"
          |
          v
[AIMD Generator Agent]
  payment-retry-prd.aimd
  ├─ :::intent      기능 목표, 성공 기준
  ├─ :::flow        사용자/시스템 흐름
  ├─ :::api         계약 초안
  ├─ :::schema      데이터 영향
  ├─ :::rules       전역 규칙
  └─ :::ai          CRITICAL, FREEZE, DONT
          |
          v
[Orchestrator / Tech Lead Agent]
  PRD 해석 + inherits 병합 + @ref 해석
  역할별 ACP 생성
  ├─ ACP-FE   -> 화면, UX 흐름, 응답 상태
  ├─ ACP-BE   -> API, 도메인 로직, 예외 처리
  ├─ ACP-DATA -> schema, migration, idempotency
  └─ ACP-QA-seed -> 테스트 기준, 성공/실패 케이스
          |
          +-------------------+-------------------+
          |                   |                   |
          v                   v                   v
[Frontend Agent]      [Backend Agent]       [Data Agent]
  UI 구현               API/서비스 구현         스키마/마이그레이션
  AIMD/ACP 갱신         AIMD/ACP 갱신           AIMD/ACP 갱신
  VERIFIED: 상태표시    VERIFIED: 에러코드      VERIFIED: 컬럼/인덱스
  OPEN: edge case       OPEN: race condition    OPEN: rollback check
          \                   |                   /
           \                  |                  /
            +-----------------+-----------------+
                              v
                    [Integration Agent]
                    병합, 충돌 확인, 계약 정렬
                    FREEZE 대상 확정
                              |
                              v
                         [QA Agent]
                         E2E, 회귀, 시나리오 검증
                         VERIFIED / OPEN 갱신
                              |
                              v
                       [Review / Deploy Agent]
                       최종 요약, 배포 준비, 릴리즈 노트
```

이 구조의 핵심은 `사람이 AIMD를 직접 손으로 작성하는 것`이 아니라, `사람이 AI를 통해 AIMD를 생성하고, 이후 여러 에이전트가 그 공통 문서를 참조·갱신하면서 필요할 때 역할별 ACP를 뽑아 쓰는 것`이다.

### 병렬 단계에서의 효과

병렬 구현 단계에서는 같은 원문을 공유하되, 에이전트마다 받아야 하는 컨텍스트가 다르다.

- 프론트엔드 에이전트는 화면 상태, 사용자 메시지, API 응답 케이스에 집중하면 된다
- 백엔드 에이전트는 도메인 로직, 재시도 정책, 에러 처리, idempotency 규칙이 중요하다
- 데이터 에이전트는 migration, 제약 조건, lock 전략, 정합성 조건이 더 중요하다

ACP를 쓰면 공통 PRD에서 필요한 부분만 추출하면서도, `CRITICAL`, `VERIFIED`, `FREEZE`, `OPEN` 같은 상태는 공통 의미를 유지할 수 있다. 즉, 역할별로 맥락은 줄이고, 규칙은 어긋나지 않게 만들 수 있다.

### 순차 단계에서의 효과

병렬 구현이 끝나면 이후 단계는 대체로 순차적이다.

- 통합 에이전트는 각 구현 결과를 맞춘다
- QA 에이전트는 확정된 사항과 미해결 사항을 분리해 검증한다
- 리뷰 또는 배포 에이전트는 최종 상태를 정리한다

이때 중요한 것은 앞 단계의 결과가 단순 로그가 아니라 `구조화된 handoff payload`로 남아 있다는 점이다.

예를 들어:

- 백엔드 에이전트가 `VERIFIED: 중복 요청 시 409 반환`을 남기면 QA는 그 조건을 다시 해석하지 않고 테스트 케이스로 바로 연결할 수 있다
- 통합 에이전트가 `FREEZE: API response contract`를 남기면 이후 프론트 수정이나 QA 보정 중 계약이 흔들리지 않는다
- 데이터 에이전트가 `OPEN: migration rollback 검증 필요`를 남기면 배포 단계까지 해당 위험이 살아남는다

### AIMD와 ACP의 역할 분담

이 시나리오에서 두 포맷의 역할은 분명하다.

- AIMD는 사람이 AI를 통해 생성한 공통 구조 문서다
- ACP는 각 단계에서 주고받는 런타임 컨텍스트다

즉, PRD 의도나 요구사항이 바뀌면 AI가 AIMD를 재생성 또는 갱신하고, 에이전트 handoff가 발생하면 ACP를 다시 빌드한다.

### 실제 예시: PRD -> ACP-FE / ACP-BE

아래는 하나의 PRD 일부가 어떻게 역할별 ACP로 분기되는지 보여주는 축약 예시다.

먼저 사람의 요청을 받아 AI가 공통 AIMD PRD를 생성한다.

```markdown
:::intent payment-retry
목표: 결제 실패 시 사용자가 즉시 재시도할 수 있어야 한다
성공 기준: 중복 결제 없이 3회까지 재시도 가능
:::

:::flow payment-retry
1. 결제 실패 -> 사용자에게 재시도 버튼 노출
2. 재시도 클릭 -> 기존 주문 컨텍스트 유지
3. 서버는 동일 주문에 대해 idempotency key 검증
4. 이미 처리된 요청이면 중복 결제 없이 기존 결과 반환
5. 실패 원인에 따라 사용자 메시지 분기
:::

:::api
POST /payments/:orderId/retry -> PaymentRetryResult
  @auth: Bearer
  @param orderId: uuid
  @ok 200: PaymentRetryResult{status,message,retryCount}
  @err 409: PAYMENT_ALREADY_PROCESSED
  @err 422: PAYMENT_RETRY_LIMIT_EXCEEDED
:::

:::ai
CRITICAL[1]: 동일 orderId에 대한 중복 결제는 절대 발생하면 안 됨
CRITICAL[2]: retryCount는 사용자에게 노출되어야 함
FREEZE: API error code contract
DONT: 결제 재시도 로직에서 새 주문 생성 금지
:::
```

이제 Orchestrator는 같은 원본에서 프론트엔드용 ACP와 백엔드용 ACP를 따로 만든다.

프론트엔드 에이전트는 전체 PRD를 다 받을 필요가 없다. 화면 상태, 사용자 메시지, 응답 필드, 수정 금지 계약만 받으면 된다.

```markdown
---
acp: "0.1"
id: payment-retry-fe-001
from: orchestrator_agent
to: frontend_agent
task: payment_retry_ui
source: docs/payment-retry.aimd
---

:::summary
결제 실패 후 재시도 UI를 구현한다.
사용자는 같은 주문 맥락에서 최대 3회까지 재시도할 수 있어야 한다.
:::

:::ai
FROM: orchestrator_agent
CRITICAL[1]: retryCount는 사용자에게 반드시 노출
CRITICAL[2]: 409 / 422 에러 메시지 UX 분기 필요
FREEZE: API error code contract
NEXT: frontend_agent
:::

:::refs
@ref:docs/payment-retry.aimd#payment-retry
@ref:src/api/payment.ts::retryPayment
:::

:::next
1. 실패 화면에 재시도 버튼 추가
2. retryCount UI 표시
3. 409, 422, 일반 실패 메시지 분기
:::
```

반대로 백엔드 에이전트는 UI 문구보다 idempotency, 예외 처리, 주문 재생성 금지 같은 제약이 더 중요하다.

```markdown
---
acp: "0.1"
id: payment-retry-be-001
from: orchestrator_agent
to: backend_agent
task: payment_retry_api
source: docs/payment-retry.aimd
---

:::summary
동일 orderId 기준 결제 재시도 API를 구현한다.
중복 결제 없이 최대 3회 재시도를 보장해야 한다.
:::

:::ai
FROM: orchestrator_agent
CRITICAL[1]: 동일 orderId 중복 결제 절대 금지
CRITICAL[2]: idempotency key 검증 필수
FREEZE: API error code contract
DONT: 새 주문 생성 금지
NEXT: backend_agent
:::

:::refs
@ref:docs/payment-retry.aimd#payment-retry
@ref:src/orders/order.service.ts
@ref:src/payments/payment.service.ts
:::

:::open
- retryCount 저장 위치 확정 필요
- 409와 기존 성공 응답 재사용 방식 정리 필요
:::

:::next
1. POST /payments/:orderId/retry 구현
2. idempotency key 검증 추가
3. 409 / 422 에러 코드 계약 준수
:::
```

QA 에이전트는 구현 자체보다 `무엇이 이미 확정되었는지`, `어떤 시나리오가 반드시 검증되어야 하는지`, `어떤 위험이 아직 열려 있는지`를 알아야 한다. 그래서 ACP-QA는 테스트 관점으로 다시 조립된다.

```markdown
---
acp: "0.1"
id: payment-retry-qa-001
from: integration_agent
to: qa_agent
task: payment_retry_validation
source: docs/payment-retry.aimd
---

:::summary
결제 재시도 기능 구현 후 검증 단계.
중복 결제 방지, retryCount 노출, 에러 코드 계약 준수가 핵심 검증 대상이다.
:::

:::ai
FROM: integration_agent
VERIFIED: 409는 이미 처리된 결제 재시도 시 반환
VERIFIED: 422는 재시도 한도 초과 시 반환
CRITICAL[1]: 동일 orderId 중복 결제 발생 여부 반드시 검증
CRITICAL[2]: retryCount 노출 및 증가 동작 검증
FREEZE: API error code contract
NEXT: qa_agent
:::

:::refs
@ref:docs/payment-retry.aimd#payment-retry
@ref:src/payments/payment.service.ts
@ref:web/src/pages/payment-failed.tsx
:::

:::open
- 동시 재시도 클릭 시 race condition 완전 차단 여부 확인 필요
- 재시도 3회 초과 후 UI 메시지 문안 최종 점검 필요
:::

:::next
1. 동일 orderId로 동시 재시도 시 중복 결제 미발생 검증
2. retryCount가 1, 2, 3으로 정확히 노출되는지 검증
3. 409 / 422 / 일반 실패 메시지와 API 응답 매핑 검증
4. 회귀 테스트에 기존 결제 성공 플로우 영향 여부 확인
:::
```

아래 표로 보면 같은 원본 PRD가 역할별로 어떻게 재구성되는지 더 빠르게 비교할 수 있다.

| 항목 | 목적 | 핵심 컨텍스트 | 대표 지시자 | 주요 다음 행동 |
|------|------|---------------|-------------|----------------|
| AIMD PRD 공통 문서 | AI가 생성한 구조 원본 유지 | 목표, 흐름, API 계약, 전역 규칙, 공통 제약 | `CRITICAL`, `FREEZE`, `DONT` | 에이전트 공통 참조 및 구조층 갱신 |
| ACP-FE | 화면/UX 구현 | 재시도 버튼, 상태 표시, 사용자 메시지, 응답 필드 | `CRITICAL[1]: retryCount 노출`, `FREEZE: API error code contract` | 실패 화면 수정, retryCount 표시, 에러 UX 분기 |
| ACP-BE | API/도메인 로직 구현 | idempotency, 중복 결제 방지, 에러 코드, 주문 재생성 금지 | `CRITICAL[1]: 중복 결제 금지`, `DONT: 새 주문 생성 금지` | retry API 구현, key 검증, 409/422 계약 준수 |
| ACP-QA | 검증/회귀 테스트 | VERIFIED 조건, OPEN 위험, 계약 유지 여부, 동시성 검증 | `VERIFIED`, `CRITICAL[1]: 중복 결제 검증`, `FREEZE: API error code contract` | 동시 재시도 테스트, retryCount 검증, 에러 매핑 및 회귀 확인 |

이 예시가 보여주는 핵심은 다음과 같다.

- PRD 원문은 하나다
- 에이전트별 payload는 다르다
- 하지만 `CRITICAL`, `FREEZE`, 계약 정보는 공통 의미를 유지한다

즉, ACP는 PRD를 조각내는 포맷이 아니라, `하나의 PRD를 역할별로 안전하게 재구성해 전달하는 포맷`이다.

### 왜 일반 Markdown PRD만으로는 부족한가

일반 Markdown PRD도 사람에게는 충분할 수 있다. 하지만 멀티 에이전트 개발에서는 아래 문제가 바로 생긴다.

- 같은 문장을 프론트와 백엔드 에이전트가 다르게 해석한다
- 어떤 요구사항이 필수인지 참고인지 구분이 흐려진다
- 누가 무엇을 검증했는지 기록이 약하다
- 병렬 작업 이후 재병합 비용이 커진다

반대로 AIMD + ACP 구조에서는:

- 사람의 의도는 하나의 AIMD 공통 문서로 정규화된다
- 역할별 컨텍스트는 분기된다
- 검증 상태와 미해결 상태가 구조적으로 남는다
- 순차 핸드오프에서 정보 손실이 줄어든다

이 때문에 `PRD 기반 병렬 개발`은 ACP/AIMD의 필요성을 설명하는 가장 강한 대표 시나리오 중 하나다.

## 도입 전략

처음부터 거대한 표준을 만들 필요는 없다. 아래처럼 단계적으로 도입하는 편이 낫다.

### 1단계: Markdown 프로파일

먼저 Markdown 기반 ACP 프로파일을 정의한다.

- 헤더
- `:::summary`
- `:::ai`
- `:::refs`
- `:::open`
- `:::next`

이 정도면 이미 대부분의 핸드오프 손실을 줄일 수 있다.

### 2단계: 정규 필드와 검증 규칙 추가

그 다음 아래를 명시한다.

- 필수 헤더 키
- 지시자 목록
- 상태값 목록
- 병합 규칙
- freeze/unfreeze 규칙

### 3단계: JSON 직렬화 프로파일 추가

프로토콜 연동이 필요해지면 JSON 직렬화를 추가한다. 이 단계에서 A2A 메시지 첨부용 구조와도 맞출 수 있다.

### 4단계: 레퍼런스 구현 공개

다음과 같은 최소 구현이 있으면 채택 장벽이 크게 낮아진다.

- Markdown ACP parser
- AIMD -> ACP builder
- ACP validator
- VS Code preview

## 기대 효과

ACP가 도입되면 다음 효과를 기대할 수 있다.

- 핸드오프 시 맥락 누락 감소
- 중요 제약 위반 감소
- 검증 완료 정보의 재계산 감소
- 에이전트 간 역할 분업 안정화
- 프로젝트 문서와 런타임 작업 사이의 연결 강화

특히 아래 효과가 크다.

> 컨텍스트가 길어지는 것이 문제가 아니라, 컨텍스트의 우선순위와 검증 상태가 불명확한 것이 문제다.

ACP는 바로 이 지점을 겨냥한다.

## 리스크와 대응

### 리스크 1. 또 하나의 문법이 되는 것 아니냐

대응:

- Markdown 기반의 얇은 프로파일로 시작한다
- 사람 중심 입력 경험을 유지한다
- AIMD 같은 AI-generated structured layer와 연결한다

### 리스크 2. 모델마다 해석이 다를 수 있다

대응:

- 지시자 수를 최소화한다
- 우선순위와 상태값을 제한된 어휘로 유지한다
- 파서와 validator를 함께 제공한다

### 리스크 3. 너무 빨리 표준화를 시도하는 것 아니냐

대응:

- 먼저 제안서와 레퍼런스 구현으로 시작한다
- 실제 핸드오프 사례를 수집해 수정한다
- 프로토콜 표준이 아니라 콘텐츠 프로파일로 위치시킨다

## 제안 문구

외부 발표나 README 첫 문단에는 아래 문구를 권장한다.

> ACP(Agent Context Payload)는 에이전트 간 핸드오프를 위해 설계된 경량 컨텍스트 포맷이다. 이 포맷은 우선순위, 검증 상태, 수정 금지 범위, 참조, 다음 행동을 구조적으로 표현하며, 사람이 AI를 통해 생성한 AIMD 같은 구조 문서에서 추출될 수 있다.

## 최종 제안

지금 필요한 것은 또 하나의 거대한 AI 표준이 아니다. 필요한 것은 에이전트 협업에서 실제로 반복되는 손실을 줄여주는 작고 명확한 공통 형식이다.

따라서 다음을 제안한다.

1. `에이전트 간 컨텍스트 페이로드 포맷`을 별도 개념으로 정의한다
2. 이를 `ACP`라는 작고 명확한 이름으로 부른다
3. Markdown 기반 최소 프로파일부터 시작한다
4. AIMD는 사람이 AI를 통해 생성하고, 여러 에이전트가 참조·갱신하는 shared structured layer로 포지셔닝한다
5. 이후 MCP/A2A 생태계와 결합 가능한 JSON 프로파일로 확장한다

이 접근은 현실적이다. 기존 도구를 버리지 않는다. 기존 Markdown 자산도 버리지 않는다. 대신 지금 비어 있는 `에이전트 핸드오프용 콘텐츠 계층`을 채운다.

그 점에서 ACP는 단순한 문서 포맷 제안이 아니라, 에이전트 협업 운영 모델의 첫 번째 실용 표준 제안이 될 수 있다.

## AI 기록의 비결정성

이 제안은 `AI가 항상 예측 가능한 문장으로 기록한다`는 가정 위에 서 있지 않다. 실제로는 같은 상황에서도 에이전트마다 요약 방식과 강조점이 달라질 수 있다.

그래서 ACP/AIMD의 핵심은 `AI가 정확히 어떤 문장을 남길지 안다`가 아니라 `AI가 어떤 종류의 기록만 남길 수 있는지 제한한다`는 데 있다.

- `VERIFIED`는 근거가 있는 확인 완료 정보만 허용
- `ASSUMPTION`은 추론이나 임시 가정만 허용
- `OPEN`은 아직 닫히지 않은 위험과 미해결 사항만 허용
- `CRITICAL[n]`은 실제 제약만 허용
- `NEXT`는 다음 작업 책임만 드러내게 함

즉, 이 구조는 문장 자체의 완전한 결정성을 보장하려는 것이 아니라 `기록의 의미 범위와 책임 경계`를 통제하려는 제안이다. 그 결과 후속 에이전트는 해석 비용을 줄이고, 사람은 실패 시점의 오판 지점을 추적할 수 있다.
