# AIMD 문서 재포지셔닝 검토보고서

## 검토 대상

- 원문: `AIMD (AI-Enhanced Markdown) v1.2(최종).md`
- 검토 질문: 이 문서를 `"에이전트 컨텍스트 페이로드 포맷 제안"`으로 재정의하면 적절한가

---

## 결론

`"에이전트 컨텍스트 페이로드 포맷 제안"`은 방향 자체는 좋습니다. 다만 현재 문서를 그 이름으로 그대로 바꾸면 범위가 어긋납니다.

현재 문서는 본질적으로 다음 두 가지가 섞여 있습니다.

1. 사람이 작성하는 Markdown 호환 소스 포맷
2. 에이전트가 소비할 컨텍스트를 추출하는 규칙

즉, 지금의 AIMD는 `payload 그 자체`라기보다 `payload를 만들기 위한 authoring format + extraction model`에 가깝습니다.

따라서 최선의 선택은 `AIMD`라는 이름을 버리고 전면 교체하는 것이 아니라, 아래처럼 계층을 분리하는 것입니다.

- `AIMD`: 작성용 소스 포맷
- `Agent Context Payload`: AIMD를 해석/병합/필터링해 에이전트에 전달하는 결과물

한 줄로 정리하면:

> `"에이전트 컨텍스트 페이로드 포맷 제안"`은 좋은 상위 프레이밍이지만, 현재 문서의 단독 새 이름으로는 부정확합니다. `AIMD = 작성 포맷`, `Payload = 산출물`로 분리하는 편이 더 설득력 있습니다.

---

## 왜 이 프레이밍이 좋아 보이는가

### 1. 문제 정의가 더 실무적으로 들립니다

`AI-Enhanced Markdown`은 새 마크업 문법처럼 들립니다. 반면 `에이전트 컨텍스트 페이로드`는 실제로 해결하려는 문제를 바로 말합니다.

- 에이전트에 어떤 컨텍스트를
- 어떤 우선순위와 계층으로
- 어떤 규칙으로 전달할 것인가

이 framing은 도구 제작자, 팀 리드, 플랫폼 설계자에게 훨씬 바로 꽂힙니다.

### 2. 저장소의 구현 방향과도 맞습니다

현재 코드에도 이미 “작성 포맷”과 “에이전트용 출력”의 분리가 존재합니다.

- `aimd-cli/src/parser.ts`: `buildForAi`, `buildForHuman`, `resolveInherits`
- `aimd-vscode/src/extension.ts`: `aimd.extractForAi`

즉 이 프로젝트는 이미 단순 문서 포맷이 아니라, `에이전트용 컨텍스트를 빌드하는 파이프라인` 쪽으로 가고 있습니다.

### 3. 토큰 절감 주장보다 운영 모델을 전면에 세울 수 있습니다

현재 문서는 토큰 절감 논리가 강합니다. 이것도 중요하지만, 더 큰 가치는 다음입니다.

- 컨텍스트 계층화
- 지시사항 상속
- agent-only / human-only 분리
- 참조 기반 컨텍스트 조립
- 재시도 감소

이건 단순한 “문서 압축”보다 `에이전트 운영 포맷`에 더 가까운 가치 제안입니다.

---

## 그대로 제목만 바꾸면 생기는 문제

### 1. `payload`라는 단어에 비해 문서가 너무 authoring 중심입니다

현재 문서의 큰 비중은 아래에 있습니다.

- `.aimd` 확장자
- CommonMark 호환
- 블록 문법
- VS Code 확장 기능
- 인간 렌더링

이건 payload wire format 설명이라기보다 `문서를 어떻게 작성하고 편집할 것인가`에 대한 설명입니다.

### 2. payload 명세로 보기에는 실행 규약이 덜 명확합니다

payload 제안으로 가려면 최소한 아래가 더 명시적이어야 합니다.

- 병합 순서
- 충돌 우선순위
- `FREEZE`와 `UNFREEZE`의 해석 규칙
- `@ref` 해석 실패 시 동작
- 순환 참조 처리
- provenance 표시
- secret/redaction 정책

현재 문서는 좋은 아이디어를 설명하지만, payload spec 치고는 normative rule이 아직 부족합니다.

### 3. `Markdown 포맷`과 `agent payload`를 같은 층위로 부르면 개념이 흐려집니다

둘은 같은 것이 아닙니다.

- 작성 포맷: 사람이 편집하는 원본
- payload: 에이전트에 실제로 주입되는 정제된 결과

이 둘을 분리하지 않으면 `:::human`은 언제 제거되는지, `inherits`는 언제 풀리는지, `@ref`는 원문인지 확장 결과인지가 모호해집니다.

## 권장 포지셔닝

가장 설득력 있는 구조는 아래입니다.

### 권장 정의

- `AIMD`: Agent-readable Markdown authoring format
- `ACP` 또는 `Agent Context Payload`: AIMD에서 빌드되는 에이전트 입력 산출물

### 권장 메시지

`AIMD는 에이전트 컨텍스트 페이로드를 안정적으로 생성하기 위한 Markdown 기반 작성 포맷이다.`

이 문장으로 가면 다음 장점이 생깁니다.

- `.aimd` 확장자와 VS Code 확장을 계속 정당화할 수 있음
- `buildForAi` 같은 구현체와 자연스럽게 연결됨
- 문서 포맷과 실행 페이로드를 구분할 수 있음
- “새 마크다운 문법 만들기”라는 거부감을 줄일 수 있음

## 문서 개편 권고

만약 이 방향으로 재작성한다면, 제목만 바꾸지 말고 문서 구조도 같이 바꾸는 편이 좋습니다.

### 1. 문서의 첫 문제 정의를 바꾸기

지금:

- Markdown 호환
- 토큰 절감
- 구조적 블록

바꾸면 좋음:

- 에이전트는 긴 산문 문서보다 구조화된 컨텍스트 payload를 더 안정적으로 소비함
- AIMD는 그 payload를 만들기 위한 소스 포맷임
- `inherits`, `@ref`, `:::ai`, `:::human`은 payload assembly 규칙임

### 2. “소스”와 “빌드 결과”를 분리해서 명시하기

예시 구조:

1. AIMD source
2. resolve inherits
3. resolve refs
4. filter by target(agent/human)
5. merge rules/ai/abbr
6. emit payload

이 파이프라인 다이어그램이 들어가면 문서의 중심축이 훨씬 분명해집니다.

### 3. payload 규약 섹션을 별도로 추가하기

최소 필요 항목:

- 입력 단위: 한 파일 또는 파일 집합
- 병합 단위: `:::ai`, `:::rules`, `:::abbr`
- 비상속 블록: `:::schema`, `:::api`, `:::flow`, `:::test`
- 우선순위: 부모 먼저, 자식 나중
- 충돌 규칙: 누적/덮어쓰기/금지 중 무엇인지
- 실패 규칙: 누락 파일, 순환 참조, 깨진 `@ref`

### 4. 도구 기능은 후반부로 내리기

현재는 VS Code 확장 설명이 문서 포맷 설명과 거의 같은 무게로 보입니다. payload framing으로 가면 툴링은 본문 후반 또는 부록으로 내리는 편이 맞습니다.

### 5. 토큰 절감은 “부가 효과”로 재배치하기

핵심 가치는 `정확한 컨텍스트 전달`이고, 토큰 절감은 그 결과 얻는 운영상 이점으로 두는 편이 더 강합니다.

## 최종 판단

검토 결과, 이 문서를 `"에이전트 컨텍스트 페이로드 포맷 제안"` 관점으로 재포지셔닝하는 것은 충분히 가치가 있습니다. 다만 `AIMD`를 완전히 대체하는 새 이름으로 쓰기보다는, `AIMD를 통해 생성되는 상위 산출물 개념`으로 두는 것이 더 정확하고 더 강합니다.

실무적으로는 아래 한 문장으로 정리하는 구성이 가장 좋습니다.

> AIMD는 사람이 유지보수할 수 있는 Markdown 호환 소스 포맷이고, 그 목적은 에이전트가 안정적으로 소비할 수 있는 컨텍스트 페이로드를 생성하는 데 있다.

이 방향으로 가면 현재 문서의 강점인 Markdown 호환성, 블록 문법, 상속, 참조, IDE 지원을 모두 살리면서도, 더 큰 비전인 `agent context operating model`로 확장할 수 있습니다.
