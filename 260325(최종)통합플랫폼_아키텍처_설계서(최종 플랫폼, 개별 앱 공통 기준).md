## 1. 결론 먼저: 왜 통합 플랫폼인가

### 분산 개발(현재) vs 통합 플랫폼(목표)

| 항목 | 분산 개발 (현재) | 통합 플랫폼 (목표) |
|------|:----------------:|:-----------------:|
| 회원가입 구현 횟수 | 앱마다 (최대 20회) | **1회** |
| 결제 연동 횟수 | 앱마다 | **1회** |
| DB 관리 | 앱마다 별도 | **Neon PostgreSQL 1개** |
| 로그인 상태 | 앱 전환 시 재로그인 | **SSO — 한번 로그인** |
| 운영 비용 | 앱수 × 인프라 비용 | **단일 인프라 (Cloudflare)** |
| 개발 속도 | 앱마다 처음부터 | **새 앱 = 비즈니스 로직만** |

### 추천 아키텍처: **단일 Next.js 앱 (Monolith with Module Boundaries)**

```
https://aiworks.kr/
├── /                          ← 통합 홈 (앱 그리드)
├── /login, /register          ← 공통 인증
├── /dashboard                 ← 사용자 대시보드
├── /apps/tax/                 ← AI-Tax
├── /apps/copywriter/          ← AI 카피라이터
├── /apps/diagnosis/           ← AI 업무진단기
├── /apps/proposal/            ← AI 제안서 빌더
├── /apps/[app-slug]/          ← 나머지 모든 앱
└── /settings, /billing        ← 공통 설정·결제
```

> **Turborepo(멀티패키지)는 아직 불필요.** 앱이 팀별로 독립 배포가 필요해질 때 검토.
> 지금은 단일 레포, 단일 배포가 개발 속도와 유지보수 모두 유리.

### 3-1. 인증 (Auth) — 최우선

**한 번 로그인하면 모든 앱에서 유지된다.**

| 기능 | 구현 방법 | 비고 |
|------|-----------|------|
| 이메일·비밀번호 로그인 | Better Auth | Edge Runtime 호환 |
| 카카오 소셜 로그인 | Better Auth OAuth2 | B2C 핵심 |
| 구글 소셜 로그인 | Better Auth OAuth2 | 선택 |
| 회원가입 | Better Auth | |
| 비밀번호 재설정 | Better Auth | |
| 세션 유지 (SSO) | Better Auth JWT | 앱 간 공유 |
| 인증 미들웨어 | Next.js middleware.ts | 보호 라우트 공통 처리 |
| 온보딩 플로우 | Next.js 페이지 | 회원가입 직후 프로필 수집 |

**파일 위치:**
```
src/app/(auth)/
├── login/page.tsx
├── register/page.tsx
├── forgot-password/page.tsx
└── onboarding/page.tsx

src/lib/auth/
├── server.ts          ← Better Auth 서버 설정
└── client.ts          ← 클라이언트 컴포넌트용

middleware.ts           ← 보호 라우트 전역 처리
```

### 3-2. 구독·결제 (Billing) — 2순위

**플랜 하나로 모든 앱의 기능을 제어한다.**

| 기능 | 구현 방법 | 비고 |
|------|-----------|------|
| 구독 플랜 관리 | 토스페이먼츠 빌링키 + subscriptions | Free/Starter/Pro/Team |
| 결제 UI | 토스페이먼츠 JS SDK `requestBillingAuth()` | 클라이언트 SDK → confirm API |
| 웹훅 처리 | 토스페이먼츠 Webhook → DB | 결제 상태 동기화 |
| 플랜별 기능 제한 | 미들웨어 + DB 체크 | 앱별 다른 제한 적용 |
| AI 사용량 추적 | api_usage_logs 테이블 | 월 토큰/호출 수 |
| 청구 내역 페이지 | Next.js 페이지 | |

**플랜 설계 (예시):**
```
Free    : 앱당 월 100,000 토큰, 기본 기능
Starter : 월 1,000,000 토큰, 모든 앱 접근, 월 ₩9,900
Pro     : 월 10,000,000 토큰, 모든 앱, 우선 처리, 월 ₩29,000
Team    : 월 50,000,000 토큰, Pro + 팀원 초대(5명), 월 ₩79,000
```

**파일 위치:**
```

├── page.tsx           ← 구독 현황·결제 UI

src/app/api/billing/
├── checkout/route.ts  ← 빌링 인증 준비 정보 반환

```

### 3-3. 공통 DB 테이블 (Neon PostgreSQL + Drizzle ORM)

**앱마다 DB를 따로 두지 않는다. 하나의 Neon 프로젝트에서 관리한다.**

#### 공통 테이블 (모든 앱이 공유)

```sql
-- Better Auth 관리 테이블 (직접 수정 금지)
"user"          -- 사용자 계정 (Better Auth)
"session"       -- 세션 토큰 (Better Auth)
"account"       -- OAuth 계정 연결 (Better Auth)
"verification"  -- 이메일 인증 (Better Auth)

-- 사용자 확장 프로필
profiles
├── id                   TEXT PK → "user".id
├── biz_name             TEXT
├── onboarding_completed BOOLEAN DEFAULT false
├── role                 TEXT NOT NULL DEFAULT 'user'  ← 'user' | 'admin' (인덱스: profiles_role_idx)
├── created_at           TIMESTAMPTZ DEFAULT now()
├── updated_at           TIMESTAMPTZ DEFAULT now()
-- ⚠️ plan 컬럼 없음 (DROP 완료): subscriptions.plan이 Source of Truth
--    플랜 조회는 반드시 getUserLimits(userId) / getActivePlan(userId) 헬퍼 사용
--    위치: platform/src/lib/apps/plan.ts

-- 구독 정보 (플랜의 단일 진실 공급원 — Source of Truth)
subscriptions
├── id                   TEXT PK
├── user_id              TEXT FK → "user"
├── toss_customer_key    TEXT           ← 토스페이먼츠 customerKey
├── toss_billing_key     TEXT UNIQUE    ← 토스페이먼츠 billingKey (자동결제용)
├── plan                 TEXT ENUM(FREE, STARTER, PRO, TEAM)  ← 유일한 plan 컬럼
├── status               TEXT           ← Stripe/Toss status 그대로 (active, canceled, past_due, trialing 등)
├── period_end           TIMESTAMPTZ
├── created_at           TIMESTAMPTZ DEFAULT now()
├── updated_at           TIMESTAMPTZ DEFAULT now()
-- ⚠️ 토스페이먼츠 웹훅 실패 시 plan 값이 stale해질 수 있음
--    웹훅 처리 실패는 반드시 Sentry에 에러 기록 + 재처리 큐 등록

-- AI API 사용량 로그 (비용 추적·제한용)
api_usage_logs
├── id              TEXT PK
├── user_id         TEXT FK → "user"
├── app_slug        TEXT       ← 어떤 앱에서 사용했는지
├── model           TEXT       ← claude-opus-4-6 등
├── input_tokens    INTEGER
├── output_tokens   INTEGER
├── cost_krw        NUMERIC    ← 원 단위 환산
├── created_at      TIMESTAMPTZ DEFAULT now()  ← 월별 집계 기준 컬럼

-- 월별 사용량 집계 성능을 위한 필수 인덱스
CREATE INDEX idx_api_usage_user_month
  ON api_usage_logs (user_id, date_trunc('month', created_at));

-- 파일 메타데이터 (실제 파일은 Cloudflare R2에 저장)
files
├── id          TEXT PK
├── user_id     TEXT FK → "user"
├── app_slug    TEXT
├── r2_key      TEXT       ← R2 오브젝트 키: {userId}/{appSlug}/{nanoid}.{ext}
├── file_name   TEXT
├── file_size   INTEGER
├── mime_type   TEXT
├── created_at  TIMESTAMPTZ DEFAULT now()   ← 파일 정렬·오래된 파일 정리용

-- 인앱 알림
notifications
├── id          TEXT PK
├── user_id     TEXT FK → "user"
├── type        TEXT       ← 'vat_deadline', 'usage_limit' 등
├── title       TEXT
├── body        TEXT
├── is_read     BOOLEAN DEFAULT false
├── created_at  TIMESTAMPTZ DEFAULT now()

-- 앱 레지스트리 (플랫폼에 등록된 앱 목록 — Stage 1.5)
app_registry
├── slug        TEXT PK              ← 고유 앱 식별자 (tax, copywriter, ...)
├── name        TEXT NOT NULL        ← 표시 이름
├── description TEXT                 ← 앱 설명
├── icon        TEXT                 ← 이모지 또는 아이콘 경로
├── category    TEXT                 ← 'business' | 'document' | 'automation' | 'utility' | 'workspace'
├── min_plan    TEXT DEFAULT 'FREE'  ← 접근 최소 플랜 (FREE | STARTER | PRO | TEAM)
├── is_active   BOOLEAN DEFAULT true ← 플랫폼 수준 on/off (false 시 미들웨어 차단 + AppGrid 미노출)
├── sort_order  INTEGER DEFAULT 0    ← 앱 그리드 정렬 순서
├── route       TEXT                 ← 앱 진입 경로 (/apps/tax)
└── created_at  TIMESTAMPTZ DEFAULT now()

### 3-4. AI API 게이트웨이 (공통 래퍼)

**모든 앱이 특정 AI 모델을 직접 호출하지 않는다. 공통 게이트웨이와 모델 라우터를 거친다.**

export async function callAIText(params: AIGatewayParams): Promise<string>

// 1. 사용자 플랜 확인 → 월 토큰 한도 초과 시 UsageLimitError throw
// 2. 질문 유형/앱 컨텍스트 기반 모델 라우팅 (quick / deep / legal / doc)

    query: `${contractType} 표준계약서 필수 조항 체크리스트`,
    category: 'contract-agreement',
    topK: 5,
  });

  // 2. 관련 법조문 실시간 조회 (law.go.kr)
  const legalArticles = await searchLegalStatute({
    query: `${contractType} 관련 법률 조문 민법 상법`,
  });

  // 3. 불공정 조항 판단 기준
  const unfairChunks = await searchStartupKB({
    query: `${contractType} 불공정 조항 약관 규제 판례`,
    category: 'compliance',
    topK: 5,
  });

  // 4. 사용자 관심 영역별 추가 검색
  const concernChunks = await Promise.all(
    concerns.map(concern =>
      searchStartupKB({
        query: `${contractType} ${concern} 주의사항 판례`,
        category: 'contract-agreement',
        topK: 3,
      })
    )
  );

  // 5. RAG 컨텍스트 조합
  return formatRAGContext({
    checklist: checklistChunks,
    legalArticles,
    unfairTerms: unfairChunks,
    concernSpecific: concernChunks.flat(),
  });
}
```

**Supabase Storage 대비 차이점:**

**R2 접근 권한 체크 패턴 (필수 준수):**

R2 키가 반드시 `{userId}/` 로 시작하는지 검증하는 헬퍼를 `lib/r2/client.ts`에 두고,
모든 API Route는 직접 `env.R2_BUCKET`에 접근하지 않고 이 헬퍼만 사용한다.

```typescript
// lib/r2/client.ts — 이 함수를 통해서만 R2에 접근
export async function getFile(userId: string, r2Key: string) {
  // r2Key가 반드시 해당 userId 소유인지 검증
  if (!r2Key.startsWith(`${userId}/`)) {
    throw new Error("Forbidden: 다른 사용자의 파일에 접근할 수 없습니다.");
  }
  // ...
}
```

> ⚠️ 새 API Route에서 파일을 다룰 때 이 패턴을 빠뜨리면 다른 사용자 파일에
> 접근 가능한 보안 구멍이 생긴다. 코드 리뷰 체크리스트에 반드시 포함할 것.

#### MCP 서버 레이어 (에이전트 시대 대비)

> AI 에이전트가 AI-Console의 비즈니스 데이터를 직접 읽고 쓸 수 있도록 MCP(Model Context Protocol) 서버를 노출한다.
> 6개월 후 에이전트 기반 워크플로가 일반화될 때, AI-Console이 에이전트의 **기억 저장소이자 실행 허브**가 된다.

```
AI-Console MCP Server
├── Resources (읽기)
│   ├── console://clients/{id}          ← 고객 정보
│   ├── console://projects/{id}         ← 프로젝트 상태
│   ├── console://tasks?assignee={id}   ← 담당 태스크 목록
│   ├── console://pipeline/summary      ← 파이프라인 요약
│   └── console://finance/monthly       ← 월 매출 현황
│
├── Tools (쓰기·실행)
│   ├── create_lead          ← 리드 추가
│   ├── update_task_status   ← 태스크 상태 변경
│   ├── log_communication    ← 소통 기록 저장
│   ├── generate_proposal    ← AI 제안서 빌더 트리거 (deeplink)

└── Prompts (에이전트 가이드)
    ├── daily_briefing       ← "오늘 할 일 알려줘"

#### 100점 스펙 — 4. DeepLink 콜백 실제 구현

> 현재 설계는 deeplink로 앱을 열기만 한다.
> 콜백을 추가해 작업 결과가 AI-Console로 자동 반환되도록 한다.

**콜백 플로우:**
```
AI-Console                    AI-Works 앱
────────────────────────────────────────────────
[제안서 생성 클릭]
     │
     ├─ deeplink 호출 ────────→ 제안서 빌더 열림
     │   ?console_callback_url   (고객 데이터 pre-fill)
     │   &console_session_id      │
     │                          [생성 완료]
     │                            │
     │ ←── POST callback ─────────┘
     │     { session_id, result_type, data }
     │
[AI-Console 자동 업데이트]
  Lead에 제안서 첨부
  상태 "제안서 발송" 으로 변경
  팀원 알림 발송
```

**콜백 API 스펙:**
```typescript
// POST /api/console/callback
// AI-Works 앱이 작업 완료 후 호출

type ConsoleCallbackPayload = {
  session_id:   string          // deeplink 시 발급한 세션 ID
  source_app:   string          // 'proposal' | 'contract' | 'content-factory' | ...
  result_type:  'proposal' | 'contract' | 'document' | 'analysis'
  tenant_id:    string
  user_id:      string
  data: {
    title:      string
    file_url?:  string          // R2 저장된 결과물 URL
    summary?:   string          // AI 요약
    metadata?:  Record<string, unknown>
  }
}

// 수신 후 처리:
// 1. session_id로 원래 컨텍스트(client_id, lead_id 등) 조회
// 2. 해당 엔티티에 결과물 자동 연결
// 3. 팀 알림 발송

#### 100점 스펙 — 5. tenant_id 멀티테넌트 설계

> Phase 1 MVP부터 `tenant_id`를 모든 테이블에 추가.
> 추후 SaaS화 시 데이터 분리 재작업 없이 바로 멀티테넌트 전환 가능.

**테넌트 구조:**
```
tenants (1개 = AI-Console을 구독하는 팀/회사)
├── id            UUID PK
├── name          TEXT              -- "홍길동 컨설팅"
├── slug          TEXT UNIQUE       -- URL slug: aiworks.kr/t/honggildong
├── plan          TEXT DEFAULT 'FREE'
├── industry_key  TEXT              -- 업종 템플릿 키
├── owner_id      TEXT FK → users.id
└── created_at    TIMESTAMPTZ

tenant_members
├── tenant_id     TEXT FK → tenants.id
├── user_id       TEXT FK → users.id
├── role          TEXT   -- 'owner' | 'admin' | 'member' | 'partner'
└── joined_at     TIMESTAMPTZ
```

**모든 console_* 테이블에 tenant_id 추가:**
```sql
-- 예시: console_clients
ALTER TABLE console_clients
  ADD COLUMN tenant_id TEXT NOT NULL REFERENCES tenants(id);

CREATE INDEX idx_console_clients_tenant ON console_clients(tenant_id);
```

**RLS 정책 (Row Level Security):**
```sql
-- 테넌트별 데이터 격리
CREATE POLICY tenant_isolation ON console_clients
  USING (tenant_id = current_setting('app.current_tenant_id'));
```

**미들웨어 테넌트 컨텍스트 주입:**
```typescript
// src/middleware.ts
// 요청마다 tenant_id를 DB 세션에 주입

const tenantId = await getTenantIdFromSession(req)
await db.execute(sql`SET app.current_tenant_id = ${tenantId}`)
```

**SaaS 전환 시 추가 작업 (Phase 5):**
```
✅ tenant_id 이미 있음 → 데이터 격리 완료
✅ RLS 이미 있음 → 보안 완료
추가 필요:
  - 테넌트별 커스텀 도메인 (aiworks.kr/t/{slug} → {slug}.aiworks.kr)
  - 테넌트별 결제 (토스 customerKey per tenant)
  - 테넌트 관리자 패널 (/super-admin)
```

## 9. Cloudflare 제약 사항 (rules.md 필수 명시)

## 10. 주의사항 및 트레이드오프

| 항목 | 장점 | 단점·주의 |
|------|------|-----------|
| 단일 DB (Neon) | 관리 용이, 비용 절감 | 한 앱 트래픽이 전체에 영향 가능 |
| 단일 Next.js 앱 | 빠른 개발, 코드 공유 | 앱 수 증가 시 빌드 시간 증가 |
| 공통 AI 게이트웨이 | 비용 추적·제한, 모델 fallback 용이 | 라우팅 버그 → 전체 앱 영향 |
| 공통 Auth (Better Auth) | UX 통일, 개발 절감 | Supabase Auth RLS 대신 수동 권한 체크 |
| Cloudflare R2 | 저렴, Workers 직접 접근 | RLS 없음, API에서 소유권 직접 체크 필수 |
| pgmq | PostgreSQL 내장, 별도 서비스 불필요 | Inngest 대비 UI/모니터링 기능 약함 |
| app_registry (Stage 1.5) | 코드 배포 없이 앱 on/off, 관리자 UI로 제어 | DB에 없는 앱은 접근 불가 — 신규 앱 추가 시 sync-registry 실행 필수 |

### A-8. AI-Mdvault ↔ 회사내부 RAG 연동

`mdvault_documents`의 문서는 내부 RAG의 **1급 지식 소스**다.
MD 파일의 `##` 섹션 구조가 RAG 청크 경계와 자연스럽게 일치한다.

| 구분 | 처리 |
|------|------|
| `is_template = true` | `{{변수}}` 빈 템플릿 → RAG 인덱싱 **제외** |
| `is_template = false` | 실제 내용 문서 → 저장/수정 시 `onSave` 훅으로 **자동 인덱싱** |

**연동 흐름:**

```
mdvault_documents 저장/수정 (is_template = false)
    └─ onSave 훅 → @ai-works/rag-engine documentIngestion.upsert()
         └─ markdown_section 청킹 (## 섹션 = 청크)
              └─ internal_chunks INSERT (tenantId, sourceType='mdvault', sourceId=doc.id)

RAG 검색 결과
    └─ metadata.sourceId → /apps/mdvault/vault/[sourceId] DeepLink (원본 문서 열기)
```

**양방향 활용:**
- **Mdvault → RAG**: 온보딩 가이드, SOP, 계약 조건 문서 저장 시 자동으로 검색 가능
- **RAG → Mdvault**: 검색 결과 컨텍스트를 기반으로 새 문서 초안 생성 (`{{변수}}` pre-fill)

## 부록 C. AI 프롬프트 스토어 — 독립 앱 설계 기준 (2026-03-25)

> **배경**: 이전에 AI-Mdvault 흡수로 결정했으나 번복. B2C 판매 구조와 B2B 내부 허브 컨셉이 AI-Mdvault(마크다운 문서 자산)와 근본적으로 다름.

### C-1. 제품 포지셔닝

```
B2C: "ChatGPT를 더 잘 쓰고 싶은 모든 사람의 첫 번째 도구"
     → 업종별 검증된 프롬프트 팩을 1~3만원에 구매 → 복사 또는 플랫폼 내 바로 실행
     → AI-Works 유료 전환의 첫 결제 트리거 상품

B2B: "AI-Works 도입 기업의 팀 내부 프롬프트 허브"
     → AI-Works Pro/Team 플랜에 포함
     → 팀원이 만든 프롬프트를 공유·버전 관리
     → 회사 내부 RAG(internal_chunks)와 연결된 전용 프롬프트
     → AI-Claworks가 "저장된 팀 프롬프트" 자동 실행 가능
```

**AI-Mdvault와의 명확한 경계:**

| 항목 | AI-Mdvault | AI 프롬프트 스토어 |
|------|-----------|----------------|
| 다루는 것 | 마크다운 **문서** 템플릿 | AI 실행용 **프롬프트** |
| 결과물 | Vault 문서 자산 | AI 응답 텍스트 |
| 수익 모델 | 플랜 구독 | 팩 단품 + 플랜 번들 |
| 타깃 | 문서 작성·관리 필요자 | ChatGPT/AI를 더 잘 쓰려는 모든 사람 |

### C-2. 핵심 페이지

| 페이지 | 경로 | 설명 |
|--------|------|------|
| 스토어 홈 | `/apps/prompt-store` | 팩 탐색, 업종 필터, 인기·신규 랭킹 |
| 프롬프트 상세 | `/apps/prompt-store/[id]` | 미리보기, 실행 모달, 구매·저장 |
| 내 라이브러리 | `/apps/prompt-store/library` | 구매한 팩, 저장한 프롬프트, 커스텀 편집 |
| 팀 허브 | `/apps/prompt-store/team` | B2B — 팀 공유 프롬프트, 역할별 접근 |

### C-3. API 구조

```
GET  /api/prompts                     ← 목록·검색·카테고리 필터
GET  /api/prompts/[id]                ← 프롬프트 상세
POST /api/prompts/execute             ← 변수 치환 후 AI 게이트웨이 실행
POST /api/prompts/save                ← 내 라이브러리 저장
GET  /api/prompts/library             ← 내 라이브러리 목록
POST /api/prompts/library             ← 커스텀 프롬프트 생성
GET  /api/prompts/team                ← 팀 허브 목록 (B2B)
POST /api/prompts/team                ← 팀 허브 프롬프트 등록
```

### C-4. DB 테이블

```sql
-- 프롬프트 카탈로그 (플랫폼 운영 + UGC Phase 2)
prompts
├── id           TEXT PK
├── pack_id      TEXT FK → prompt_packs
├── title        TEXT NOT NULL
├── description  TEXT
├── category     TEXT          ← 업종/기능 분류
├── content      TEXT NOT NULL ← {{변수}} 포함 프롬프트 원문
├── variables    JSONB         ← [{ key, label, placeholder }]
├── tags         JSONB
├── is_public    BOOLEAN DEFAULT true
├── rating_avg   NUMERIC DEFAULT 0
├── use_count    INTEGER DEFAULT 0
└── created_at   TIMESTAMPTZ DEFAULT now()

-- 프롬프트 팩 (판매 단위)
prompt_packs
├── id           TEXT PK
├── name         TEXT NOT NULL
├── description  TEXT
├── category     TEXT
├── price_krw    INTEGER DEFAULT 0   ← 0 = 무료
├── prompt_count INTEGER DEFAULT 0
├── is_active    BOOLEAN DEFAULT true
└── created_at   TIMESTAMPTZ DEFAULT now()

-- 사용자 라이브러리 (구매·저장·커스텀)
user_prompt_library
├── id           TEXT PK
├── user_id      TEXT FK → "user"
├── prompt_id    TEXT            ← null이면 커스텀 프롬프트
├── title        TEXT NOT NULL
├── content      TEXT NOT NULL   ← 커스텀 편집 버전
├── is_custom    BOOLEAN DEFAULT false
└── saved_at     TIMESTAMPTZ DEFAULT now()

-- 프롬프트 평가
prompt_ratings
├── id           TEXT PK
├── prompt_id    TEXT FK → prompts
├── user_id      TEXT FK → "user"
├── rating       INTEGER         ← 1~5
├── body         TEXT
└── created_at   TIMESTAMPTZ DEFAULT now()

-- B2B 팀 허브
team_prompt_hubs
├── id           TEXT PK
├── org_id       TEXT NOT NULL   ← AI-Console tenant_id 연계
├── name         TEXT NOT NULL
└── created_at   TIMESTAMPTZ DEFAULT now()

team_prompt_members
├── hub_id       TEXT FK → team_prompt_hubs
├── user_id      TEXT FK → "user"
├── role         TEXT DEFAULT 'member'   ← 'admin' | 'member'
└── joined_at    TIMESTAMPTZ DEFAULT now()
-- PRIMARY KEY (hub_id, user_id)
```

### C-5. AI-Claworks 연동 포인트

```
AI-Claworks (카카오톡 인터페이스)
  └─ "사업계획서 제출용 요약 만들어줘"
       └─ AI-Console MCP → team_prompt_hubs 조회
            └─ "사업계획서 요약" 팀 프롬프트 발견
                 └─ POST /api/prompts/execute 자동 호출
                      └─ 결과를 카카오톡으로 반환
```

### C-6. 개발 순서

```text
Step 1. 스토어 홈 + 프롬프트 상세 + 바로 실행 모달 MVP
Step 2. 내 라이브러리 저장 및 커스텀 편집
Step 3. 프롬프트 팩 토스페이먼츠 구매 연동
Step 4. 팀 허브 (B2B) — 팀 공유·역할 관리
Step 5. 회사 내부 RAG + AI-Claworks 연동
Step 6. UGC — 사용자 프롬프트 등록·평가 (Phase 2)
```

## 부록 B. 글로벌 AI 채팅 어시스턴트 앱별 통합 로드맵 (2026-03-25)

> 채팅 인프라는 완성됐다. 이제 각 앱이 완성될 때마다 아래 체크리스트를 따라 채팅을 "활성화"한다.
> **두 가지만 하면 된다**: `usePlatformChat`에 currentData 추가 + `tools.ts`에 툴 1개 등록.

### B-1. 앱별 채팅 통합 체크리스트

| 앱 | 1단계: context 주입 | 2단계: 툴 등록 | 완료 기준 |
|----|-------------------|----------------|-----------|
| AI-Mdvault | `currentDocument` (열린 문서 내용) | `search_templates` ✅ 이미 완료 | Vault POST 완성 후 |
| AI-Contract | `currentContract` (분석 중인 계약서) | `analyze_contract` | 실 AI API 완성 후 |
| PDF Editor Pro | `currentPdf` (열린 PDF 메타데이터) | `analyze_pdf`, `sign_request` | PDF Vault 완성 후 |
| AI-Tax | `currentReportPeriod` (처리 중인 기간) | `classify_expense`, `calc_vat` | E2E 검증 완료 후 |
| AI-Analytics | `currentDataset` (분석 중인 파일명) | `run_analysis` | R2 업로드 완성 후 |
| AI-Proposal | `currentProposal` (작성 중인 제안서) | `refine_proposal` | 내보내기 완성 후 |
| AI-Word | `currentDocument` (편집 중인 문서) | `edit_document` | API·DB 연결 완성 후 |
| AI-Sheet | `currentSheet` (편집 중인 시트명) | `query_sheet` | API·DB 연결 완성 후 |
| AI-ContentFactory | `currentBrand` (선택된 브랜드 프로필) | `generate_content` | 캘린더 완성 후 |

### B-2. 단계별 실행 계획

#### 1단계: 지금 — 각 앱 핵심 기능 완성에 집중

우선순위 순서:

```text
1. AI-Mdvault   → Vault 저장 POST + DB 테이블 생성
2. AI-Contract  → 실 AI 분석 API + DB 마감
3. AI-Tax       → 업로드-분류-리포트 E2E 검증
4. AI-Word      → 문서 저장 API + DB 연결
5. AI-Sheet     → 스프레드시트 저장 API + DB 연결
```

#### 2단계: 앱 완성 직후 — 채팅 연결 (앱당 1~2시간)

각 앱이 완성되는 즉시 아래 두 작업을 추가한다:

```typescript
// ① 앱 Client에 currentData 추가
usePlatformChat({
  appSlug: "contract",
  appName: "AI-Contract",
  currentView: screen,
  // 앱 완성 후 추가:
  currentDocument: currentContractData,
})

// ② tools.ts에 앱 전용 툴 1개 추가
const analyzeContract: PlatformTool = {
  name: "analyze_contract",
  displayLabel: "계약서 분석",
  description: "현재 열린 계약서를 분석하고 리스크 조항을 요약합니다.",
  // ...
}
```

#### 3단계: 앱 전체 안정화 후 — 깊은 통합

- `chat_messages` DB 테이블 생성 → 대화 히스토리 저장
- 크로스앱 오케스트레이션 (예: "이 계약서 내용으로 제안서 초안 만들어줘")
- AI-Console MCP와 채팅 어시스턴트 연결
